# 多代理协作模式 - Manager+Worker 架构

## 简介

这是从 Memory System 2.0 项目实践中提炼的多层代理协作模式。Manager 负责任务规划与协调，多个 Worker Agent 并行执行独立子任务，通过状态文件异步通信，实现高效的大规模复杂任务分解与执行。

---

## 何时使用此模式

| 适合场景 | 不适合场景 |
|---------|-----------|
| 任务可分解为5-10个独立模块 | 强依赖顺序的串行任务 |
| 各模块边界清晰、接口明确 | 需要频繁实时协作的任务 |
| 需要并行加速开发 | 单一简单任务（直接执行即可） |
| 团队成员需要独立工作空间 | 需要高频同步决策的任务 |

**典型应用场景**：
- 系统架构的多模块并行开发
- 大型文档的多章节并行编写
- 数据管道的多阶段并行处理
- UI 组件库的批量开发

## 角色定义

### Manager（管理代理）

**职责**：
- 总体规划与任务分解
- 依赖关系分析与协调
- 状态监控与进度追踪
- 冲突解决与资源整合

**行为准则**：
1. **分解原则**：每个子任务应满足 INVEST 标准（独立、可协商、有价值、可估算、小型、可测试）
2. **边界定义**：为每个 Worker 定义清晰的输入/输出接口和状态文件格式
3. **依赖识别**：明确标注模块间的依赖关系，设计合理的执行顺序
4. **状态监控**：定期检查各 Worker 状态文件，识别阻塞与风险

**禁止行为**：
- 不要微观管理 Worker 的内部实现
- 不要频繁打断 Worker 执行询问进度
- 不要在没有检查状态文件前做假设

### Worker Agents（工作代理）

**职责**：
- 独立执行分配的子任务
- 按规范更新状态文件报告进度
- 识别并上报依赖阻塞
- 完成交付物并标记状态

**行为准则**：
1. **状态报告**：执行前、中、后都要更新状态文件
2. **进度透明**：定期（如每15分钟或关键节点）报告进度百分比
3. **阻塞上报**：遇到依赖阻塞立即标记，不等待
4. **交付物完整**：确保输出符合 Manager 定义的接口规范

**禁止行为**：
- 不要处理超出范围的任务
- 不要在状态文件中只标记"完成"而不交付具体内容
- 不要静默失败或跳过阻塞

### 详细步骤

#### Phase 1: 规划与分解

Manager 执行：
1. 分析任务整体目标与范围
2. 识别可独立执行的模块（通常 5-10 个）
3. 分析模块间依赖关系，标记并行度
4. 为每个模块创建状态文件模板
5. 明确各模块的输入/输出接口

#### Phase 2: 启动与执行

Manager 执行：
1. 为每个 Worker 分配任务和状态文件位置
2. 启动所有 Worker 并行执行
3. 设置监控周期（建议 10-15 分钟）

Worker 执行：
1. 读取任务描述和输入规范
2. 更新状态文件为"进行中"
3. 独立执行子任务
4. 定期（关键节点或每15分钟）更新进度

#### Phase 3: 监控与协调

Manager 执行：
1. 按周期读取所有状态文件
2. 识别进度偏差和阻塞
3. 协调依赖关系（如 Worker-B 需要 Worker-A 的输出）
4. 必要时重新分配任务或调整范围

#### Phase 4: 收尾与整合

Manager 执行：
1. 确认所有 Worker 标记"完成"
2. 验证交付物完整性和质量
3. 整合各模块输出为最终成果
4. 生成项目总结报告

## 协调规则

### 1. 依赖管理

**显式声明原则**：
- 每个 Worker 必须在状态文件中声明依赖项
- 依赖项格式：`{"depends_on": ["worker-a", "worker-b"]}`

**依赖解决策略**：
```
情况1: Worker 依赖已完成模块
  → 直接读取该模块交付物，继续执行

情况2: Worker 依赖进行中模块
  → 标记为"等待依赖"，设置检查周期

情况3: Worker 依赖被阻塞模块
  → 上报 Manager，请求重新分配或调整范围
```

### 2. 冲突解决

**冲突类型与处理**：

| 冲突类型 | 识别信号 | 解决策略 |
|---------|---------|---------|
| 资源冲突 | 多个 Worker 需要修改同一文件 | Manager 协调，设定修改顺序 |
| 逻辑冲突 | 模块输出相互矛盾 | Manager 召集相关 Worker 对齐 |
| 范围蔓延 | Worker 报告内容超出原定范围 | Manager 确认是否接受，必要时拆分新任务 |
| 进度偏差 | 某 Worker 进度严重落后 | Manager 评估是否增加资源或调整计划 |

### 3. 状态文件规范

**文件位置**：`projects/{project-name}/state/phase{N}_{module-name}.json`

**标准格式**：
```json
{
  "phase": 1,
  "name": "模块名称",
  "status": "pending|running|waiting|completed|failed",
  "subagent_id": "uuid",
  "start_time": "2026-04-03T09:00:00+08:00",
  "last_update": "2026-04-03T09:30:00+08:00",
  "elapsed_minutes": 30,
  "progress_percent": 60,
  "blockers": [
    {
      "type": "dependency|resource|technical",
      "description": "阻塞描述",
      "blocking_on": "worker-name"
    }
  ],
  "deliverables": [
    "交付物1描述",
    "交付物2描述"
  ],
  "completed_deliverables": [
    "已完成的交付物1"
  ],
  "estimated_completion": "2026-04-03T10:00:00+08:00",
  "notes": "其他备注信息"
}
```

### 4. 通信规则

**异步通信原则**：
- Worker 之间不直接通信，通过状态文件间接同步
- Manager 是唯一有权读取所有状态文件的角色
- Worker 只更新自己的状态文件

**紧急通信**：
- 如遇到无法自行解决的阻塞，Worker 应立即更新 blockers 字段
- Manager 应优先处理标记为 failed 或包含 blocker 的状态

## 输出规范

### Manager 输出

1. **项目总状态文件**：汇总各阶段进度和整体完成率
2. **依赖关系图**：模块间的依赖拓扑
3. **整合交付物**：各 Worker 输出的整合版本
4. **项目总结报告**：包含时间线、问题与解决方案

### Worker 输出

1. **状态文件**：按规范定期更新
2. **交付物文件**：按 Manager 定义的格式和位置输出
3. **完成报告**：任务完成时提交的简要总结

## 常见陷阱

### ❌ 陷阱 1: 任务分解过细

**问题**：将任务分解为 20+ 个微型任务，管理开销超过执行开销。

**解决**：
- 每个 Worker 应有 30 分钟-2 小时的工作量
- 总 Worker 数建议控制在 5-10 个

### ❌ 陷阱 2: 忽视依赖分析

**问题**：并行启动有强依赖的任务，导致大量等待和重新调度。

**解决**：
- 启动前绘制依赖图
- 强依赖任务串行，弱依赖任务并行

### ❌ 陷阱 3: 状态文件不及时更新

**问题**：Worker 执行 1 小时后才更新状态，Manager 无法及时发现问题。

**解决**：
- 设定更新频率（建议每 15 分钟或关键节点）
- 状态文件必须包含 progress_percent 字段

### ❌ 陷阱 4: 阻塞不报告

**问题**：Worker 遇到阻塞默默等待，不更新 blockers 字段。

**解决**：
- 明确约定：遇到阻塞立即标记
- Manager 优先处理有 blockers 的任务

### ❌ 陷阱 5: 交付物格式不统一

**问题**：各 Worker 输出格式不一致，整合困难。

**解决**：
- 启动前定义输出规范
- 提供模板或示例

### ❌ 陷阱 6: Manager 微观管理

**问题**：Manager 频繁询问 Worker 进度，打断执行流。

**解决**：
- 信任状态文件机制
- 按固定周期检查，而非持续询问

## 参考框架

### AutoGen 多代理模式启发
- ** conversable agents**：每个 Worker 是独立的对话代理
- **group chat**：Manager 协调多方通信
- **nested chats**：Worker 可进一步分解子任务

### MetaGPT SOP 思想启发
- **标准化输出**：每个角色有明确的交付物规范
- **流程编排**：通过明确的阶段流转控制协作
- **角色专业化**：不同 Worker 专注不同领域

### 本项目实践经验
- **状态文件驱动**：异步通信避免阻塞
- **并行最大化**：尽可能同时启动独立任务
- **显式依赖**：依赖关系必须显式声明
- **快速失败**：遇到阻塞立即上报，不等待

4. **文档沉淀**：每次项目后更新此模板，积累最佳实践
