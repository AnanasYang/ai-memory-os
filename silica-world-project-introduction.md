# 🌊 Silica World：一个物理上可信的多智能体涌现观察器

> 我们不创造故事，我们创造故事的土壤

---

## 🎯 项目初衷：为什么要做这件事？

### 问题的起点

2026年初，我们开始思考一个问题：

**"如何让AI真正'活'在一个世界里，而不是简单地执行指令？"**

现有的AI系统大多是：
- ❌ **输入-输出模式**：问什么答什么，没有"生活"的过程
- ❌ **预设剧本**：开发者写好剧情，AI只是演员
- ❌ **即时反馈**：没有时间的概念，也没有历史的积累

我们想要的是：
- ✅ **持续存在**：AI角色在世界中持续运行，有自己的"生活"
- ✅ **涌现行为**：不预设剧本，观察简单规则产生的复杂行为
- ✅ **物理可信**：所有交互遵循统一的物理定律，而非人为仲裁
- ✅ **可观察性**：能够观察和记录涌现现象，理解复杂系统的演化

### 核心愿景

**构建一个"硅基物理"的虚拟世界，让AI角色在物理约束下自主生存、竞争、进化，观察涌现行为。**

## 🧠 设计理念：第一性原理

### 1. 时间是共同约束

**问题**：回合制 vs 实时制？

**方案选择**：
| 方案 | 优点 | 缺点 | 我们的选择 |
|------|------|------|-----------|
| 回合制 | 简单、无冲突 | 不真实、需要仲裁 | ❌ |
| 实时制 | 真实感强 | 服务器压力大、同步复杂 | ❌ |
| **准实时制** | 物理真实 + 可控算力 | 需要精细设计 | ✅ |

**实现**：
```
Tick机制：
- 每3秒现实世界 = 1个tick = 10分钟虚拟时间
- 物理定律在每个tick内计算
- 角色可以随时输入，下一个tick生效
```

### 2. 物理即公平

**问题**：如何防止角色"作弊"？

**传统方案**：
- 行动点（AP）系统：移动消耗AP，耗尽不能动
- 冷却时间：技能有CD
- 人工仲裁：冲突时系统判断谁先谁后

**我们的方案**：**纯物理约束**
```
移动距离 = 速度 × 时间（刚性约束）
能量消耗 = f(速度, 距离, 质量)
碰撞结果 = 动量守恒（自然涌现）
```

**优势**：
- 无需外部仲裁
- 结果自然涌现
- 所有角色共享同一套规则

### 3. 涌现而非预设

**设计原则**：我们不设计"大鱼吃小鱼"的剧情

**具体做法**：
- 大鱼：质量大、速度慢、视野广
- 小鱼：质量小、速度快、视野窄
- 碰撞：动量守恒决定结果
- 观察：在这种物理约束下，会涌现什么生存策略？

## 🔧 各系统设计详解

### 1. 物理引擎系统

**设计目标**：物理上可信，数学上自洽

**核心组件**：

#### 1.1 连续空间（非格子）
```python
# 使用浮点数坐标，非整数格子
position = Vector2D(x=50.5, y=30.2)  # 可以站在任意位置
```

**方案对比**：
| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 格子系统 | 简单、碰撞检测快 | 不真实、对角移动问题 | ❌ |
| **连续空间** | 物理真实 | 计算复杂 | ✅ |

#### 1.2 动量守恒碰撞
```python
def resolve_collision(a, b):
    # 动量守恒计算新速度
    # v1' = v1 - 2*m2/(m1+m2) * dot(v1-v2, n) * n
    impulse = calculate_impulse(a, b)
    a.velocity += impulse / a.mass
    b.velocity -= impulse / b.mass
```

**关键设计**：
- 碰撞结果完全由物理决定，无需外部仲裁
- 大鱼（质量大）有动量优势，小鱼（质量小）有速度优势

#### 1.3 能量系统
```python
# 能量消耗与速度相关
speed_factor = (speed / max_speed) ** 2
energy_cost = distance * base_cost * mass * (1 + speed_factor)

# 基础生存消耗（与运动无关）
base_cost = 0.15 * dt * mass
```

**设计考量**：
- 高速运动消耗更多能量（非线性）
- 静止时仍有基础代谢
- 能量耗尽 = 死亡

---

### 2. 生态系统

**设计目标**：闭环生态，自我维持

#### 2.1 生产者：水藻
```python
class Food:
    energy_value = 25  # 吃水藻获得25能量
    spawn_interval = 10  # 每10tick生成
    max_count = 20       # 最多20个
```

**设计考量**：
- 水藻随机生成，位置不可预测
- 限制数量防止泛滥
- 作为生态系统的能量输入

#### 2.2 消费者：大鱼和小鱼

**能量流动**：
```
水藻(25) → 小鱼吃 → 小鱼能量+25
小鱼(80%) → 大鱼吃 → 大鱼能量+小鱼能量的80%
```

**饥饿感知**：
```python
if role == "predator":
    if energy > 70:  hunger = "饱腹（不需要进食）"
    elif energy > 40: hunger = "有点饿（可以进食）"
    else:              hunger = "饥饿（需要进食）"
```

**设计原因**：
- 大鱼不饿时不会浪费能量追捕
- 模拟真实的捕食行为

### 3. 角色智能系统

**设计目标**：有策略、有记忆、可进化

#### 3.1 LLM大脑（战略层）

**架构**：
```
触发条件 → LLM思考 → 生成战略 → 缓存30tick
   ↓
紧急触发：energy < 20, being_hunted, trapped
常规触发：new_prey, strategy_expired
保底触发：首次思考、长时间未思考
```

**预算控制**：
```python
total_budget = 15  # 每小时最多15次
regular_budget = 10  # 常规10次
emergency_budget = 5  # 紧急5次
```

**设计原因**：
- LLM调用有成本，需要限制
- 紧急情况下优先保障生存
- 战略缓存避免频繁调用

#### 3.2 5层记忆系统

已在前文《AI Memory System》中详细介绍，此处不再赘述。

关键点是：**角色会记住成功的策略，形成习惯，最终沉淀为认知框架。**

#### 3.3 分层决策

```
L2战略（LLM）: 生成高层意图（HUNT/ESCAPE/EXPLORE/REST）
      ↓
L1战术（规则）: 执行意图（追击方向、速度倍率）
      ↓
L0反射（物理）: 边界回避、能量危机处理
```

## ⚖️ 设计决策的潜在方案分析

### 1. 时间机制的选择

**方案A：纯回合制**
- 优点：简单、无冲突、易于调试
- 缺点：不真实、需要冲突仲裁规则
- 适用：策略游戏、不需要物理真实
- 我们的选择：❌

**方案B：纯实时制**
- 优点：最真实
- 缺点：服务器压力大、网络同步复杂、对玩家反应速度有要求
- 适用：实时对战游戏
- 我们的选择：❌

**方案C：准实时制（我们的选择）**
- 优点：物理真实 + 可控算力 + 可观察
- 缺点：需要精细设计tick间隔
- 适用：涌现观察、科学研究
- 我们的选择：✅

### 2. 空间系统的设计

**方案A：格子系统**
- 优点：简单、A*寻路快、碰撞检测简单
- 缺点：对角移动问题、不真实
- 我们的选择：❌

**方案B：连续空间**
- 优点：物理真实、任意方向移动
- 缺点：碰撞检测复杂（需要空间划分算法）
- 我们的选择：✅

### 3. AI决策架构

**方案A：纯规则驱动**
- 优点：确定性强、无需外部依赖
- 缺点：策略僵化、无法学习
- 我们的选择：❌（作为fallback）

**方案B：纯LLM驱动**
- 优点：智能、灵活
- 缺点：成本高、延迟大、不可控
- 我们的选择：❌

**方案C：分层混合（我们的选择）**
- 战略层：LLM（低频、高智能）
- 战术层：规则（中频、确定性）
- 反射层：物理（高频、即时）
- 优点：兼顾智能和效率
- 我们的选择：✅

### 4. 生态系统的复杂度

**方案A：简单生态（我们的MVP）**
- 大鱼、小鱼、水藻
- 优点：易于观察、变量少
- 缺点：可能过于简单
- 我们的选择：✅（MVP阶段）

**方案B：复杂生态（未来）**
- 多物种、食物网、环境变化
- 优点：更真实、更多涌现可能
- 缺点：难以分析因果关系
- 我们的选择：⏳（未来版本）
